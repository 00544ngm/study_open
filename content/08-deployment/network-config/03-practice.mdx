## 实战一：从零配置服务器网络

### 场景

你有一台全新的阿里云 ECS（Ubuntu 24.04），公网 IP 为 `114.55.111.133`，需要在上面部署一个 Next.js 应用（端口 3000）和一个 FastAPI 服务（端口 8000）。

### 要求

1. 配置 OS 级防火墙，只放行 SSH、HTTP、HTTPS 以及 3000、8000 端口
2. 配置阿里云安全组，同样只放行上述端口
3. 确保两个服务可以通过外网访问
4. 防火墙规则重启后不丢失

### 参考实现

```bash
# 第一步：配置 ufw 防火墙
ufw default deny incoming
ufw default allow outgoing
ufw allow ssh
ufw allow 80/tcp
ufw allow 443/tcp
ufw allow 3000/tcp
ufw allow 8000/tcp
ufw enable
ufw status verbose

# 第二步：验证端口监听
ss -tlnp | grep -E '3000|8000'

# 第三步：如果是 Docker 部署，确认端口映射
docker ps --format "table {{.Names}}\t{{.Ports}}"

# 第四步：验证外网连通性
curl -s http://114.55.111.133:3000
curl -s http://114.55.111.133:8000
```

### 思考题

1. 为什么需要同时配置安全组和 OS 防火墙？只配一个行不行？
2. 如果服务监听在 `127.0.0.1`，即使防火墙和安全组都放行了，外网能访问吗？
3. 如何临时放行一个 IP 进行调试？

## 实战二：HTTPS + 域名 + 反向代理

### 场景

为你的 AI Agent 平台配置：

```
用户 → https://ai-agent.example.com → Nginx (SSL) → http://localhost:3000 (Next.js)
```

### 要求

1. 注册域名并配置 A 记录
2. 申请 Let's Encrypt 证书
3. 配置 Nginx 反向代理 + SSL 终止
4. HTTP 自动跳转到 HTTPS
5. 验证 SSL 评级（A+）

### 参考实现

```nginx
# /etc/nginx/sites-available/ai-agent
server {
    listen 443 ssl http2;
    server_name ai-agent.example.com;

    ssl_certificate     /etc/letsencrypt/live/ai-agent.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/ai-agent.example.com/privkey.pem;

    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers on;

    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 10m;

    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    add_header X-Content-Type-Options nosniff;
    add_header X-Frame-Options DENY;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
    }
}

server {
    listen 80;
    server_name ai-agent.example.com;
    return 301 https://$host$request_uri;
}
```

```bash
# 验证 SSL 配置
curl -sI https://ai-agent.example.com | grep -i strict-transport-security

# 使用 SSL Labs 测试（浏览器打开）
# https://www.ssllabs.com/ssltest/analyze.html?d=ai-agent.example.com
```

### 思考题

1. 如果同时有多个子域名（api.、app.、admin.），如何用一张证书覆盖所有子域名？
2. HSTS header 的作用是什么？配置后如果证书过期了会怎样？
3. 为什么 Nginx 配置了 `proxy_set_header X-Forwarded-Proto $scheme`？

## 实战三：Docker 多服务网络搭建

### 场景

使用 Docker Compose 部署一个完整的 AI Agent 应用栈：

```
Nginx（443/80）→ Next.js（3000）+ FastAPI（8000）+ PostgreSQL（5432）+ Redis（6379）
```

### 要求

1. Nginx 对外暴露 80 和 443 端口
2. Next.js 和 FastAPI 只在内网通信，不暴露到宿主机
3. PostgreSQL 和 Redis 完全隔离，仅后端服务可访问
4. 服务间通过容器名（service name）通信
5. 配置健康检查

### 参考实现

```yaml
# docker-compose.yml
services:
  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx.conf:/etc/nginx/conf.d/default.conf
      - ./ssl:/etc/letsencrypt
    depends_on:
      - frontend
      - backend
    networks:
      - public
    restart: unless-stopped

  frontend:
    build:
      context: ./frontend
      dockerfile: Dockerfile
    expose:
      - "3000"
    environment:
      - NEXT_PUBLIC_API_URL=https://api.example.com
    depends_on:
      - backend
    networks:
      - public
      - internal
    restart: unless-stopped

  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    expose:
      - "8000"
    environment:
      - DATABASE_URL=postgresql://user:pass@db:5432/agent
      - REDIS_URL=redis://redis:6379
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_started
    networks:
      - internal
      - private
    restart: unless-stopped

  db:
    image: postgres:16-alpine
    expose:
      - "5432"
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: agent
    volumes:
      - pgdata:/var/lib/postgresql/data
    networks:
      - private
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U user -d agent"]
      interval: 5s
      timeout: 5s
      retries: 5
    restart: unless-stopped

  redis:
    image: redis:7-alpine
    expose:
      - "6379"
    volumes:
      - redisdata:/data
    networks:
      - private
    restart: unless-stopped

volumes:
  pgdata:
  redisdata:

networks:
  public:
    driver: bridge
  internal:
    driver: bridge
  private:
    driver: bridge
```

```nginx
# nginx.conf
upstream frontend_upstream {
    server frontend:3000;
}

upstream backend_upstream {
    server backend:8000;
}

server {
    listen 80;
    server_name example.com;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl http2;
    server_name example.com;

    ssl_certificate     /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    # Frontend
    location / {
        proxy_pass http://frontend_upstream;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # API
    location /api/ {
        proxy_pass http://backend_upstream;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

### 验证

```bash
# 启动所有服务
docker compose up -d

# 查看网络隔离效果
docker compose exec frontend ping backend  # 应该通（同一网络）
docker compose exec frontend ping db       # 应该不通（不同网络）

# 验证服务连通性
curl -s http://localhost/api/health

# 查看日志
docker compose logs -f
```

### 思考题

1. 三层网络隔离（public / internal / private）的设计目的是什么？
2. 如果 frontend 也需要连接 Redis，网络拓扑应该怎么调整？
3. 健康检查的 `interval` 和 `retries` 参数在实际部署中应该如何设置？

## 实战四：故障排查演练

### 场景

你收到报警：用户无法访问 `https://api.example.com`。服务端 `docker ps` 显示容器在运行，`curl localhost:8000` 返回正常。

### 排查步骤

```bash
# 1. 检查端口监听
ss -tlnp | grep 8000

# 2. 检查防火墙
iptables -L INPUT -n -v | grep 8000
ufw status

# 3. 检查 Nginx 配置
nginx -t
systemctl status nginx

# 4. 检查 Nginx 日志
tail -50 /var/log/nginx/error.log
tail -50 /var/log/nginx/access.log

# 5. 从本机测试公网端口
curl -s -o /dev/null -w "%{http_code}" http://localhost:8000
curl -s -o /dev/null -w "%{http_code}" http://127.0.0.1:8000

# 6. 测试安全组（在外网机器执行）
telnet api.example.com 443

# 7. 检查 DNS 解析
dig +short api.example.com
```

### 常见故障原因速查

| 症状 | 最可能的原因 | 优先级 |
|------|------------|--------|
| `connection refused` | 服务没监听或端口不对 | 最高 |
| `connection timed out` | 防火墙/安全组拦截 | 最高 |
| `no route to host` | 网络不通、IP 错误 | 高 |
| `502 Bad Gateway` | 后端没启动或 Nginx 配置错误 | 高 |
| `504 Gateway Timeout` | 后端响应超时 | 中 |
| SSL 证书错误 | 证书过期或域名不匹配 | 中 |
| DNS 解析不到 IP | DNS 配置未生效或 TTL 未过 | 低 |

### 思考题

1. `connection refused` 和 `connection timed out` 的区别是什么？各自反映什么问题？
2. 如何用一条命令持续监控端口是否在线？
3. 如果用户在中国访问海外服务器很慢，有哪些优化手段？
