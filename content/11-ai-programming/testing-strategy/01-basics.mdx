## 核心

### 为什么测试对 AI 代码更重要

AI 生成代码的"黑盒"特性意味着你无法通过阅读来完全确认正确性。测试是唯一的兜底网。核心理念：**先写测试，再让 AI 生成实现**。

### 测试金字塔

```
    ╱╲
   ╱ E2E ╲       ← 少量，关键路径（~5%）
  ╱────────╲
 ╱  集成测试 ╲    ← 中等，API/DB 交互（~15%）
╱─────────────╲
╱   单元测试    ╲   ← 大量，每段 AI 代码的兜底网（~80%）
╱───────────────╲
```

### AI 代码的测试流程

1. **先写接口定义和类型签名**
2. **再写测试用例**（这些测试就是给 AI 的 prompt）
3. **让 AI 生成实现**
4. **运行测试验证**

```python
# Step 1: 定义接口
def process_order(user_id: int, items: list[dict]) -> OrderResult:
    """处理用户订单。"""
    ...

# Step 2: 写测试（这就是 prompt）
"""
请实现 process_order 函数，以下是测试用例：

def test_process_order_normal():
    result = process_order(1, [{"id": "A1", "qty": 2}])
    assert result.success is True
    assert result.total > 0

def test_process_order_empty():
    result = process_order(1, [])
    assert result.success is False
    assert "empty" in result.error
"""

# Step 3: AI 生成实现
# Step 4: pytest 验证通过 ✓
```

### 测试类型详解

```python
# 1. 单元测试：测单一函数
def test_calculate_total():
    assert calculate_total([{"price": 10, "qty": 2}]) == 20
    assert calculate_total([]) == 0
    assert calculate_total([{"price": -1, "qty": 1}]) == 0  # 负数处理

# 2. 集成测试：测模块交互
def test_order_pipeline():
    user = create_test_user()
    result = place_order(user.id, [{"id": "P1", "qty": 1}])
    assert result.order_id is not None
    assert get_inventory("P1") == 0  # 库存扣减

# 3. 契约测试：测 API 接口
def test_api_contract():
    response = client.post("/api/orders", json={"user_id": 1, "items": [...]})
    assert response.status_code == 201
    assert "order_id" in response.json()
```

### 测试覆盖门禁

```bash
# 运行测试并检查覆盖率
pytest --cov=src/ --cov-fail-under=80

# AI 代码必须跑通全部测试才能合并
# 覆盖率下降也会被拒绝
```

### 关键原则

**测试通过 = 代码可用**。对于 AI 生成代码，这个等式比手写代码更成立——因为你没有仔细读过 AI 的每一行实现，但你信任测试的验证结果。
